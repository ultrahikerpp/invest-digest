### 各主題重點


#### 主題一：當 Windows 遇上 CUDA，AI PC 如何權力重組？



- 作者認為，NVIDIA 在 Computex 發布的 RTX Spark 平台（N1/N1X 晶片）短期內對財報幾乎沒有貢獻，因為筆電要到秋天才上市，初期出貨量小、定價高、目標客群窄，真正意義在於 PC 產業「時代分水嶺」的長線佈局。

- 作者認為，微軟與 NVIDIA 形成高度互補的雙向需要：微軟需要 NVIDIA 來對抗 Apple Silicon，NVIDIA 需要 Windows 作為唯一具備規模的消費端舞台，因為 macOS 不開放、Linux 消費端市佔低於 5%。

- 作者認為，NVIDIA 真正的長線戰略是在 Windows 之上建立「由 CUDA 定義的 AI 計算層」，使其成為所有人都繞不過的「水電公司」，複製其在資料中心的生態滲透模式。

- 作者認為，AI 時代 PC 產業正從「鬆散模組化」走向「協同設計」，高階 AI PC 要求硬體、作業系統與軟體棧一起設計，蘋果的垂直整合已是典範，Windows 陣營正跟進。

- 作者認為，RTX Spark 採用的統一記憶體架構（最高 128GB LPDDR5X）短期仍屬極客與專業開發者的小眾市場，對記憶體市場的實質貢獻是 2027、2028 年之後的種子，當前記憶體價格循環仍由資料中心 HBM 主導。



#### 主題二：當 AI 走向「代理人」時代，Intel 如何用 CPU 奪回系統調度權？



- 作者認為，Intel 在 Computex 2026 的核心論述是：隨著 AI 從「單次問答」進化到「多代理人協作（Agentic AI）」，CPU 在系統中的「編排層（Orchestration Layer）」角色正在回歸，因為 Agentic 任務高度依賴分支邏輯、記憶體管理與任務調度，而非 GPU 擅長的矩陣運算。

- 作者認為，Intel 透過 Perplexity 展示的「端雲協同」架構（敏感資料在 Core Ultra 本地處理、深度推理送上雲端）和與 SambaNova 的異質運算合作，具體呈現了 Xeon 作為「系統指揮官」的價值定位。

- 作者認為，Intel 的戰略核心在於「不論你用誰的加速器，你都需要 Xeon 作為最完美的黏著劑」，主動承認異質運算為常態，反而擴大了自身的適用場景。

- 作者認為，x86 與 Arm 架構之爭無法在 Computex 得出答案，因為兩家展示的均為對自家架構最有利的場景；真正的勝負需觀察後續企業與 OEM 的實際出貨量、軟體生態相容性，以及雲端服務商的資本支出比例。

- 作者認為，18A 製程落地是 Intel 編排層論述的重要加分項，但市場態度是「先看到持續交付再說」，因為 Intel 的 roadmap 執行歷史讓市場保持謹慎。



#### 主題三：從回購王變成融資王，Google 為什麼突然募資 850 億美元？



- 作者認為，「Alphabet 融資是為了幫員工繳稅」是斷章取義，融資結構清楚區分：約 550 億美元用於 AI 基礎建設與算力擴張，約 300 億美元處理 RSU 稅務；關鍵是 Alphabet 選擇保留自由現金流完整性，並全額給付員工股票以留住 AI 頂尖人才。

- 作者認為，這筆融資最有趣之處在於展示了已上市科技巨頭的「光速吸金」優勢：從宣布到資金入帳僅五天，而 OpenAI、Anthropic 等未上市公司的融資之路動輒耗時數月，這種流動性護城河本身就是競爭壁壘。

- 作者認為，Berkshire Hathaway 認購 100 億美元是重要信號，代表即使在最保守的價值投資標準下，Alphabet 的 AI 基礎建設投資仍被認為具備長期 ROIC 吸引力與安全邊際。

- 作者認為，這筆融資正在「逼迫所有人提前攤牌」：Alphabet 提前鎖定稀缺晶片與電力資源，先動手者取得最便宜的資金，動作慢的競爭對手將面對更挑剔的投資人與更貴的融資成本。

- 作者認為，我們正處於 AI 的「重工業階段」，物理基建（算力、電力、資料中心）必須先行，公開市場超額認購、Berkshire 錨定投資與評級機構正面評估，共同為這個週期投下信任票。



#### 主題四：拒絕賺快錢，從 Build 2026 看微軟的策略



- 作者認為，Nadella 明確拒絕「嫉妒型創新」，微軟不加入「誰的模型最聰明」軍備競賽，而是聚焦在「微軟能獨特地做什麼」，即成為企業 AI 生態的作業系統平台，如同 PC 時代的 Windows、雲端時代的 Azure。

- 作者認為，微軟的核心競爭框架是「爬山機（Hill-climbing）」：最關鍵的護城河不是基礎模型的強弱，而是企業能否把自身的隱性知識、專屬資料與工作流程持續餵給模型，讓模型在特定業務場景中不斷優化。

- 作者認為，Build 2026 所有發表（MAI 模型、Frontier Tuning、Microsoft IQ、Scout、Solara、MXC 沙盒）本質上都是為了讓企業「安全、規模化、受治理地運作自己的爬山機」，是一套完整的企業工具箱而非零散技術更新。

- 作者認為，微軟正從「重度依賴 OpenAI 單一合作夥伴」轉向「建立重要但非獨占的夥伴關係」，MAI 自研模型與多模型選擇反映此戰略轉變；Nadella 明確表示不把算力大量押注在少數模型公司，而是分配給超大規模雲端、自身應用業務與自研模型研究三個長期桶子。

- 作者認為，Agent 自主運作帶來的「按人頭計費失效」是根本性商業難題，微軟的解法是「基本人頭費＋彈性算力消耗費」混合定價，但 Scout 仍在小規模驗證階段，市場接受度尚未明朗，短期內仍需以 Azure 成長品質與早期 Agent 採用跡象來評估進展。



#### 主題五：Nvidia 對記憶體需求減半？



- 作者認為，「Nvidia 記憶體需求腰斬」是嚴重的去脈絡化：被降配的是 CPU 側的系統記憶體（LPDDR5X），而非 GPU 側的 HBM4，HBM4 規格與需求完全未受影響。

- 作者認為，Nvidia 降配 LPDDR5X 是純粹的供應鏈務實決策：高密度 16 層堆疊模組良率受限導致卡供應，改用 96GB 成熟模組可加速出貨、每台機架省下 80 萬美元，且 SOCAMM2 插槽設計保留了未來現場升級的彈性。

- 作者認為，從總量邏輯看，單機記憶體配置減半意味著相同產能可組裝雙倍機架數量，記憶體廠商的產能依然滿載，總位元需求並未縮水，且未來客戶回頭採購 192GB 升級模組將帶來「售後拉貨潮」。

- 作者認為，此事件反而加速 AI 記憶體三階層架構演進：HBM4（熱層）、LPDDR5X SOCAMM（溫層）、高性能 SSD（冷層），NAND Flash 在 Agent 與長文本時代因 KV Cache 需求而戰略地位上升，同時擁有 DRAM 與 NAND 的廠商（Samsung、SK hynix、Micron）因此具備更完整的 exposure。

- 作者認為，記憶體超級週期的邏輯沒有改變，瓶頸只是在不同層級間移動，總需求方向清晰；面對「XX 需求腰斬」類標題，應先追問「腰斬的是哪一層、為什麼、對其他層有何影響」。

### 核心觀點



- 作者認為，在市場波動時，投資者首先應釐清自己的「投資身分」——若是交易者則嚴守紀律止損，若是投資者則應深度體檢部位的集中度、信念感、估值安全邊際與因果邏輯，而非被「事後諸葛」式的市場噪音左右；作者本人的選擇是硬扛、收縮衛星標的、加碼高信念核心持倉。

- 作者認為，我們正處於 AI 的「重工業階段」，物理基建必須先行，所有超大型科技公司都在加速搶佔算力、電力與晶片資源，公開市場的「光速吸金」能力（如 Alphabet 五天完成 850 億美元融資）本身就是一道無法逾越的護城河，也是為何獨角獸需要盡快上市的核心原因。

- 作者認為，NVIDIA 在 AI 時代的終極戰略是「在每一層都成為繞不過的基礎設施」——在雲端是資料中心 GPU，在企業端是 Agentic AI 的異質運算調度者，在消費端則是以 CUDA 為核心的 PC 計算層；這是一場以 CUDA 生態系滲透為主軸、需要數年觀察的長線佈局。

### 創作者建議的觀察方向


**AI PC / NVIDIA RTX Spark**



- 開發者是否開始習慣在 Windows 筆電上直接調用本地端 TensorRT 進行日常開發

- 微軟 Copilot 與各類 AI Agent 是否在搭載 NVIDIA 晶片的系統上運行最流暢

- 「軟硬一體」體驗是否逐漸拉開與傳統 x86 筆電的差距，進而重塑消費者購買決策



**Intel vs. Arm 架構之爭**



- 企業與 OEM 的實際拉貨與出貨量

- 企業將現有 IT 基礎設施遷移至 Arm 架構時的實際阻礙與隱性成本

- x86 陣營執行新一代 AI Agent 軟體時的實際調度優勢表現

- 雲端服務商的資本支出與採購比例變化



**微軟 Build 2026 策略**



- Azure 成長品質與內部應用算力使用效率

- Scout 等早期 Agent 產品的企業採用跡象

- 混合定價模式（人頭費＋消費費）的市場接受度，以及何時對財報產生實質貢獻



**Nvidia Vera Rubin 記憶體架構**



- 高密度 16 層堆疊 LPDDR5X 模組的良率與產能改善進度

- 客戶未來是否回頭採購 192GB 模組進行二次升級（「售後拉貨潮」）

- NAND Flash 在 Agent / 長文本時代的需求成長速度

- AI 三階層記憶體架構（HBM4 / LPDDR5X / SSD）的軟體調度成熟度
